## What "Neural Rendering" Actually Means

Neural rendering is a broad term for any rendering technique that uses a trained neural network as part of the image production process. It doesn't mean the whole frame is generated by AI — it means a neural network is inserted at some stage in the pipeline to do work that would otherwise be impossible or prohibitively expensive with traditional algorithms.

DLSS is the most widespread example: instead of a hand-written upscaling algorithm, a convolutional neural network performs the upscaling. The network was trained offline on pairs of low- and high-resolution images, and it learned what high-resolution detail *should* look like given a low-resolution input plus temporal history.

---

## DLSS: How the Neural Upscaler Works

DLSS (Deep Learning Super Sampling) uses a **convolutional neural network (CNN)** to reconstruct a high-resolution image from a lower-resolution rendered frame plus historical data.

The inputs the network receives each frame:
- Current low-resolution color buffer (the rendered frame)
- Motion vectors (per-pixel screen-space displacement from last frame)
- Depth buffer
- Exposure / brightness metadata
- Camera jitter offset
- A reprojected copy of the previous high-resolution output (history)

The network outputs a high-resolution image that incorporates detail inferred from both the current frame and accumulated history.

**How the CNN is structured:** DLSS uses a U-Net style architecture — a type of CNN with an encoder path that compresses spatial resolution to extract abstract features, and a decoder path that reconstructs full resolution. Skip connections between encoder and decoder allow the network to blend fine spatial detail with high-level context. The network runs as a compute shader on Tensor Cores.

**Training:** NVIDIA trained DLSS offline using a dataset of ultra-high resolution ground-truth images rendered with supersampling (many samples per pixel). The network learned to predict what a high-resolution pixel *should* look like given low-resolution input and motion history. This is a supervised learning setup — the network minimizes the difference between its output and the ground-truth high-res image.

**Why this matters for my capstone:** Because DLSS was trained on clean, well-behaved inputs, it performs well when motion vectors are accurate and temporal history is stable — and degrades when they're not. The network has no fallback for bad data; it just produces artifacts. This is why my architectural patterns exist: I'm keeping the inputs clean so the network can do its job.

## DLSS Versions and the Evolution of the Approach

- **DLSS 1.0:** Fully neural, but relied on a single static network. Quality was inconsistent and it required per-game training. Largely considered a failed experiment.
- **DLSS 2.0:** Major redesign. Used a generalized temporal network trained across many games. Introduced the motion vector + history accumulation approach. This is the version that actually worked.
- **DLSS 3 / Frame Generation:** Added an entirely new component — a separate AI model that *generates* new frames between rendered frames using **optical flow**.
- **DLSS 4 / Multi Frame Generation:** Blackwell-exclusive. Can generate up to three synthetic frames for every one rendered frame. Strictly an inference workload — runs entirely on Tensor Cores.

---

## Optical Flow and Frame Generation

Frame generation is architecturally different from upscaling. Upscaling takes one low-res frame and makes it high-res. Frame generation takes two *full* rendered frames (frame N and frame N+1) and synthesizes one or more new frames between them.

**Optical flow** is the technique used to analyze motion between frames. It produces a per-pixel flow field — for each pixel in frame N, where did it end up in frame N+1? With this flow field, the generator can interpolate pixel positions at the midpoint between the two frames, producing a plausible intermediate image.

The optical flow analysis in DLSS 3+ runs entirely on the GPU using a dedicated Optical Flow Accelerator (OFA) present in Turing and later Nvidia architectures. It's not a traditional motion vector calculation done by game code — it's done by the GPU scanning the actual rendered pixel buffers.

**Performance model:** If a game renders at 60 FPS and frame generation produces one synthetic frame per rendered frame, the display sees 120 FPS. The game's *simulation* still runs at 60 FPS — only the display output doubles. This means the input latency doesn't actually halve; DLSS Frame Generation uses NVIDIA Reflex to mask the latency increase.

**Constraints frame generation adds:**
- Frame pacing must be extremely consistent. If rendered frames arrive unevenly, the synthesized frames have inconsistent timestamps and the result feels wrong.
- The game must not be CPU-bound. If the CPU can't submit frames consistently, frame generation can't help.
- UI must be rendered *after* frame generation, at full resolution, or it gets synthesized incorrectly (text smears, crosshairs ghost).

These are new architectural constraints that didn't exist before frame generation. They explain why workload budgeting and fixed-timestep simulation matter even more in the frame generation era.

## On-Device Inference: Tensor Cores and How Neural Networks Run in Real Time

Running a neural network during a game frame sounds impossibly expensive. It works because of dedicated hardware: **Tensor Cores** (Nvidia) are matrix multiplication units designed specifically for the kind of math CNNs require.

A neural network inference pass is fundamentally a series of matrix multiplications. A Tensor Core can perform a 4×4 matrix multiply in a single clock cycle, and hundreds of them operate in parallel. This is why inference that would take milliseconds on a general CUDA shader core takes microseconds on Tensor Cores.

**Precision formats matter:** DLSS runs inference in FP16 (half-precision float) or INT8 (8-bit integer) rather than FP32. Lower precision = smaller matrices = faster matrix multiply. Blackwell adds FP4 support, which is why it can run DLSS 4's multi-frame generation at practical performance levels.

**Memory bandwidth:** Neural networks are memory-bandwidth bound more than compute-bound during inference. The weights of the network have to be loaded from VRAM each frame. This is why GPU VRAM bandwidth improvements (like GDDR7 in RTX 50 series) directly improve neural rendering performance, not just traditional rendering.

## Training vs. Inference — Why the Distinction Matters

This is a distinction I kept seeing in documentation without a clear explanation, so I specifically asked about it.

**Training** is the process of teaching the neural network. It runs offline (not during gameplay) on massive datasets, typically across thousands of GPU-hours on data center hardware. The network adjusts its internal weights using backpropagation — a process of computing gradients through the network and nudging weights toward lower error. NVIDIA trains DLSS on their own high-end cluster; game developers don't touch this process.

**Inference** is using the trained network to produce outputs. This is what runs in real time during gameplay. The weights are fixed — inference is just a forward pass through the network: feed in inputs, get outputs. No learning happens.

**Why this matters for my project:**
- The constraints on game code come from *inference*, not training. The network was trained to expect clean inputs; if inference receives noisy inputs, it produces visually wrong outputs.
- Game developers interact with the inference side — they integrate the DLSS SDK, provide the required buffers, and tune parameters. They have no visibility into the training data or network architecture.
- The network is essentially a black box from the game developer's perspective. You can't fix artifacts by tweaking the network — you fix them by improving the inputs. This is the entire premise of my capstone.

## AMD FSR: How It Compares

FSR (FidelityFX Super Resolution) takes a different approach:

- **FSR 1** was a pure spatial upscaler — no neural network, no temporal component. Fast and simple, but quality was noticeably lower than DLSS.
- **FSR 2/3** added temporal accumulation (similar to DLSS 2) but uses hand-written algorithms rather than a neural network. It still requires motion vectors, depth, and jitter.
- **FSR 4** (released 2025) finally introduced a neural network component, putting it closer to DLSS architecturally.

FSR's open-source nature means the algorithm is inspectable — I can read what it actually does with motion vectors, depth, and reactive masks. This has been useful for understanding the concrete requirements my patterns need to satisfy, since FSR's documentation is more transparent than DLSS's about its internal logic.

## Emerging Area: AI-Assisted Animation and NPC Behavior

Beyond upscaling, neural networks are being applied to other rendering-adjacent systems. This is more forward-looking but directly relevant to the "AI-driven GPU architecture" angle of my research paper.

**Physics-based character animation:** Projects like Nvidia's PhysX and learned motion matching use neural networks to blend animation states in real time, producing more natural movement than traditional state machines. The network takes character state as input and outputs joint positions/rotations.

**NPC behavioral inference:** Rather than hand-authored behavior trees, some research systems use lightweight neural networks for NPC decision-making. The network runs as a GPU compute shader, allowing many NPCs to be evaluated in parallel.

**ACE (Avatar Cloud Engine):** Nvidia's in-progress system for neural NPC dialogue and behavior. Runs inference on Tensor Cores to drive conversational NPC responses in real time.

These systems add another layer of GPU compute workload, which is why workload budgeting becomes even more critical — the GPU is now handling rendering, upscaling, frame generation, *and* behavioral inference simultaneously.

## Connection to the Blackwell Architecture

This is where the hardware research in my paper connects to the software patterns:

- Blackwell allocates significantly more die area to Tensor Cores than Ada Lovelace
- It introduces FP4 precision support, making very large neural networks feasible at inference time
- The 5th generation Optical Flow Accelerator is more capable and lower-latency
- Multi Frame Generation (generating 3 synthetic frames per rendered frame) is only possible on Blackwell because prior architectures couldn't run the optical flow model fast enough to maintain low latency

This is the hardware-software co-design argument: Blackwell wasn't designed to run existing workloads faster — it was designed to enable a *new class* of workloads (multi-frame generation, real-time neural animation) that didn't exist before. And those new workloads impose new constraints on game code. That chain — hardware → features → software constraints → architectural patterns — is the core argument of my capstone.

## Key Takeaways

- DLSS is a CNN trained offline to reconstruct high-res images from low-res inputs + motion history. It requires clean, accurate inputs to produce clean outputs.
- Frame generation uses optical flow to synthesize intermediate frames — it adds strict frame pacing requirements on top of upscaling requirements
- Temporal instability in game code (bad motion vectors, non-deterministic simulation) doesn't just look bad — it actively misleads neural models that were trained to expect well-behaved inputs
- Tensor Cores make neural inference fast enough for real time; FP16/INT8 precision keeps it within frame budgets
- From a game developer's perspective, the neural network is a black box. The only lever you have is the quality of the inputs — which is exactly what my architectural patterns are designed to control
- Blackwell's expanded AI silicon is what makes multi-frame generation possible, and multi-frame generation is what makes the software constraints in my project more consequential than ever
